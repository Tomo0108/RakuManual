## 1. 課題の再定義

### 表面的な課題

フローを途中で修正すると、マニュアル項目が増える・項番がズレる・削除済み手順が残るなど、前後の整合を取るのが難しい。

### 現状（コードベース）

| 層 | フロー変更時の挙動 |
| --- | --- |
| DeepDiveItem | `confirmFlow` で再構築（既存保持、ラベル変更で `recheck`、削除ノードは落ちる） |
| ManualSection | **自動再同期なし**。再生成ボタン／セクション単位モック再生成のみ |
| 項番 | `assignSectionNumbers` はフロー側。マニュアル側 `sectionNumber` は追従しない |
| 紐づけキー | `stepId`（= FlowNode.id）と `sectionNumber` |

主要パス: `app/src/features/flow/FlowEditorTab.tsx`, `app/src/features/manual/ManualTab.tsx`, `app/src/features/deepdive/DeepDiveTab.tsx`, `app/src/lib/types.ts`

### 本当に解くべき痛み（否定派との合意点）

| 誤解しやすい目標 | 実際の痛み |
| --- | --- |
| フローとマニュアルを常に完全一致させる | 変更後に「どこを見直せばよいか分からない」不安 |
| 自動でマニュアルを直す | 自分の編集・承認済み文言が壊される恐怖 |
| 不整合ゼロのデータモデル | 意図的な差分（まとめ説明・注意事項・省略）を許容できない窮屈さ |

**結論:** 今解くべきは「整合性エンジン」ではなく **変更後の不安を減らし、既存文書資産を壊さずに見直し導線を提供すること**。

要件定義書も「各工程のタブ間を自由に行き来でき、後戻りによるデータ消失は発生させない」と明記している。自動上書きやロックで探索を殺す設計は要件と衝突する。

## 2. 問題の分解

| ID | 問題 | 危険度 | 備考 |
| --- | --- | --- | --- |
| P1 | 追加ステップ → マニュアルに未反映 | 高 | 深掘りには出るが ManualSection が無い |
| P2 | 削除ステップ → マニュアルに残骸 | **最高** | 存在しない手順が成果物に残る |
| P3 | 項番ドリフト | 中〜高 | 表示ズレ。自動再採番は人間の参照を壊しうる |
| P4 | ラベル変更 → タイトル／本文が古い | 中 | 深掘りは `recheck`、マニュアルは未対応 |
| P5 | 分岐・順序変更 → 本文の関係性が無効化 | 高 | セクション単位では検知しづらい |
| P6 | 承認済み／手編集本文の破壊 | **最高** | 無差別再生成・AI自動マージで最悪化 |
| P7 | 部分再生成の影響範囲が不明 | 中 | 「どこまで直すべきか」が分からない |
| P8 | 後からフロー更新→マニュアルも再生成したい | 高 | 必要な更新と残したい箇所が混在する |
| P9 | 再生成・編集後に戻せない | **最高** | 現行は `version` カウンタと操作ログのみで復元不可 |

## 3. ブレスト結果の要約

### 3-A. 提案派が挙げた手段

| 案 | 要点 | 侵襲度 |
| --- | --- | --- |
| 単一ソース・オブ・トゥルース | フロー／深掘りを正、マニュアルは派生ビュー | 中〜高 |
| 差分パッチ同期 | 確定時に ManualSection へ差分適用＋プレビュー | 中 |
| ドリフト検知 UI | ズレをバッジ／一覧で可視化 | 低〜中 |
| 不変 stepId + リベース | 本文は ID に紐づけ、構造だけ載せ替え | 中 |
| イベントソーシング | 変更履歴から整合状態を計算 | 高 |
| ロック／再確定ゲート | マニュアル生成後のフロー編集に確認を挟む | 低〜中 |

推奨ハイブリッド（提案派）: 不変 ID + リベース基盤 × 差分サマリー × ドリフトバッジ × 承認済み本文保護。

### 3-B. 否定派の主要批判

| 対象 | 批判の核心 |
| --- | --- |
| SSOT | フローとマニュアルは責務が違う。従属化は編集価値を破壊する |
| 差分同期エンジン | 分割・統合・紐づけ曖昧で Google Docs マージ問題の再実装になる |
| ドリフト検知 alone | 解決導線のない警告はノイズ製造機 |
| イベントソーシング | プロトタイプには過剰。検証が遅れる |
| ロック | 探索的な業務整理 UX を潰す |
| 全再生成 | 承認・レビュー資産を壊し「怖くて押せない」になる |
| AI 自動マージ | もっともらしく壊す。削除は見逃されやすい |
| 項番自動再採番 | 口頭参照・レビューコメント・研修対応を壊す |

**やらない方がいいこと（否定派リスト・採用）**

1. ManualSection をフローから常時自動再生成する
2. 既存本文を AI が勝手に上書きする
3. `sectionNumber` を同期キーにする
4. フロー変更のたびに項番を自動振り直す（表示の追従と破壊的再採番は別）
5. 「不整合」を全部エラー扱いする
6. 差分同期エンジン／イベントソーシングを先に作る
7. AI 自動マージをデフォルトにする
8. 編集ロックで行動を制限する
9. ズレを許容できないデータモデルにする
10. 検知だけして解決導線のない警告 UI

### 3-C. 統合判断

| 採用 | 保留／縮小 | 却下（現段階） |
| --- | --- | --- |
| 影響可視化（要確認バッジ・見直し候補一覧） | 構造リベース（追加は候補、削除は隔離） | 完全 SSOT |
| 本文は触らない（提案は横並び） | 項番の表示追従（安定 ID と分離した上で） | イベントソーシング |
| `stepId` を安定キー、`sectionNumber` は表示 | 軽い確認ゲート（ソフト警告） | ハードロック |
| 「意図的差分」状態 | 分岐変更の粗いフラグ | AI 自動マージ／自動削除 |
| 新規は「未配置候補」、削除は「廃止候補」 | — | 全再生成デフォルト |

## 4. 目指す体験（成功の定義）

> フローを直してもマニュアルが壊れた感じがしない。どこを見直せばよいか一目で分かり、自分の文章は勝手に消えない。

### 定性

- 追加ステップが「未配置候補」として見える
- 削除ステップの残骸が「廃止候補」として隔離される
- 手編集／承認済み本文は保護される
- 「再生成すべき箇所」と「そのままでよい箇所」が区別できる
- 意図的にフローと一致させない章を「意図的差分」にできる

### 定量（プロトタイプ計測）

- フロー変更後の `needs_review` / `orphaned` / `unplaced` 件数
- 全体再生成ボタンの使用回数（減るほど良い）
- ユーザーが手動で項番を直した回数（減るほど良い）
- エクスポート前の見直し候補の未解消数
- 提案文の部分採用率（AI 提案を入れるフェーズ以降）

## 5. 検証仮説と Kill Criteria

本格実装の前に、以下を最小 UI で検証する。

| 仮説 | 最小検証 | 成功条件 | 捨てる条件 |
| --- | --- | --- | --- |
| H1: 求めているのは自動同期ではなく見直し箇所の提示 | 変更 stepId に紐づくセクションへ「要確認」バッジ | 迷わず修正に進める | 「結局分からない／邪魔／全部手で見る」 |
| H2: 自動反映より反映候補がよい | 追加・修正・削除を候補枠で提示。本文は未変更 | 採用／無視を判断できる | 毎回無視される |
| H3: 項番は安定していた方が安心 | 既存番号維持、新規は末尾または未配置 | 「番号がズレないので安心」 | 「毎回綺麗に振り直してほしい」が明確多数 |
| H4: ズレはエラーではなく状態で十分 | `ok` / `needs_review` / `intentional_difference` / `orphaned` / `unplaced` | 状態で文書整理できる | 状態操作自体が使われない |
| H5: 再生成は上書きではなく草案なら許容 | 既存横に AI 提案 | 部分採用される | 信用されず全文手直しの方が早い |

## 6. 目標アーキテクチャ（軽量）

### 原則

1. **フローは構造の正、マニュアルは説明文書の正**（互いの完全従属はしない）
2. **紐づけは不変 `stepId`。項番は人間向け表示**
3. **自動でやるのは状態付けまで。本文更新はユーザー操作**
4. **追加は候補、削除は隔離、マージは提案**
5. **意図的差分を第一級状態として許す**

### データ拡張（案）

`ManualSection`（または付随メタ）に最小追加:

| フィールド | 意味 |
| --- | --- |
| `stepId` | 既存。安定キー |
| `syncStatus` | `ok` \| `needs_review` \| `intentional_difference` \| `orphaned` \| `unplaced` |
| `sourceSnapshot` | 生成／最終同期時の label・kind・分岐要約の簡易スナップショット |
| `lastReviewedFlowVersion` | 任意。プロトではフロー確定カウンタでも可 |

DeepDive 既存の `recheck` と用語を揃えると学習コストが下がる。

### フロー確定後の処理パイプライン（目標）

```text
Flow confirm
  → DeepDive 再構築（現行）
  → ManualImpact 計算（追加／削除／ラベル差分／分岐差分）
  → ManualSection の syncStatus 更新のみ（本文は触らない）
  → 「見直し候補 N 件」サマリー表示
```

本文への構造リベース（空セクション追加・orphan 隔離）は Phase B。自動上書きは対象外。

## 7. フェーズ別開発計画

### Phase A — 影響可視化 MVP（先にやる）

**目的:** 仮説 H1 / H4 を検証。コード侵襲を最小化。

| ID | 成果物 | 内容 |
| --- | --- | --- |
| A1 | `computeManualImpact(flow, manual)` | stepId 存在・ラベル差分・orphan・missing を算出 |
| A2 | マニュアル画面バナー | 「見直し候補 N 件」＋フィルタ |
| A3 | セクションバッジ | `要確認` / `廃止候補` / `未配置` / `意図的差分` |
| A4 | アクション（最小） | 「このまま（意図的差分）」「要確認を解除」「編集へ」 |
| A5 | モックシナリオ | フロー追加／削除／改名後の整合崩れを再現する検証プロジェクト |

**やらない:** 本文自動更新、項番自動振り直し、ロック、AI マージ。

**完了条件:** フロー変更 → マニュアルタブで影響が一覧でき、本文が一切変わっていないこと。

---

### Phase B — 構造の安全なリベース

**目的:** P1 / P2 を壊さずに緩和。仮説 H2 / H3。

| ID | 成果物 | 内容 |
| --- | --- | --- |
| B1 | 未配置候補トレイ | 新規 step → ManualSection は自動挿入せず候補として提示。挿入位置はユーザー選択 |
| B2 | orphan 隔離 | 削除 step のセクションは消さず「廃止候補」へ。残す／廃止／別 step に再紐づけ |
| B3 | 項番ポリシー | 既定は既存番号安定。表示上の推奨番号はフローから算出して「提案」として見せる（破壊的再採番はオプトイン） |
| B4 | 差分サマリーモーダル | フロー確定直後に追加／削除／改名件数を短く提示 |

**やらない:** 分割・統合の自動解決、分岐本文の自動書き換え。

**完了条件:** 追加で項番が勝手に崩れない。削除残骸がエクスポート前に必ず候補として残る。

---

### Phase C — 選択的再生成（保持選択）

**目的:** 後からフローを更新したあと、マニュアルも更新したいが、**残したいセクションは明示的に守る**。

| ID | 成果物 | 内容 |
| --- | --- | --- |
| C1 | 反映ウィザード | 「フロー変更をマニュアルに反映」開始時に、セクションごとの操作を選択 |
| C2 | 保持選択 | 各セクション: `keep`（残す）/ `regenerate`（再生成）/ `archive`（廃止候補のまま） |
| C3 | 既定値 | 承認済み・手編集は `keep`、要確認・新規は `regenerate` を初期選択。ユーザーが上書き可 |
| C4 | 草案確認 | `regenerate` 対象はいきなり上書きせず草案→適用（仮説 H5） |
| C5 | 適用前スナップショット | 変更対象の現行版を必ず版履歴に保存してから適用 |
| C6 | 下流ゲート | Export / Overview で未解消 orphan・needs_review を警告 |

**やらない:** 全セクション強制再生成、保持確認なしの一括上書き、削除の自動実行。

---

### Phase D — マニュアル版管理・復元

**目的:** 適宜もとに戻せる。要件 F-6「差分・復元」をプロトタイプで検証可能な形にする。

| ID | 成果物 | 内容 |
| --- | --- | --- |
| D1 | セクション版履歴 | 編集・承認・再生成・復元のたびにリビジョンを保存 |
| D2 | 履歴 UI | セクション単位で過去版一覧・簡易差分・ワンクリック復元 |
| D3 | 一括反映の復元点 | 選択的再生成の適用直前に「復元ポイント」を作成し、まとめて戻せる |
| D4 | 公開版との分離（段階導入） | まずは編集中の版履歴。公開版ピン留めは永続化後に本格化 |

---

### Phase E — 将来（必要が証明されてから）

| 候補 | 着手条件 |
| --- | --- |
| 本格差分パッチエンジン（分割・統合） | Phase B 後も分割／統合の手作業が明確なボトルネックになったとき |
| フロー自体の版管理 | マニュアル版とセットで監査が必要になったとき |
| ソフト確認ゲート（ハードロックではない） | 意図せずフローを壊す事故が実測で多いとき |
| イベントソーシング全面採用 | 履歴自体が事業価値の中核になったときのみ |

## 9. リスクと緩和

| リスク | 緩和 |
| --- | --- |
| バッジ過多で無視される | 件数サマリー＋重要度（orphan > ラベル変更）。一括「意図的差分」は慎重に |
| 「同期していない」と不満 | コピーで「本文は保護しています。見直し候補を確認してください」と明示 |
| 項番方針のユーザー分裂 | H3 を先に検証。振り直しは明示操作のみ |
| 構造リベースが本文位置を壊す | Phase B は自動挿入せず候補トレイ |
| スコープ膨張 | 本計画の「やらないこと」を PR／実装チェックリストに固定 |
| 選択再生成で保持を忘れる | 承認済み・手編集は既定 `keep`。確定前に保持件数を再表示 |
| 版が増えすぎる | 自動保存は「意味のある区切り」（再生成適用・承認・復元・明示保存）に限定 |

## 10. 推奨ロードマップ（技術的順序）

```text
Phase A（影響可視化）
  └─ 検証: H1, H4
       ├─ 失敗 → 検知 UX を捨て、より粗い「フロー変更後はマニュアル全文レビュー」案内に退行
       └─ 成功 → Phase B（候補トレイ・orphan 隔離・項番安定）
            └─ Phase D-min（セクション版の保存／復元の受け皿）
                 └─ Phase C（保持選択付きの選択的再生成＋草案適用）
                      └─ Phase D（差分 UI・一括復元ポイント）
                           └─ なお不足 → Phase E を個別に再評価
```

**順序の意図:** 再生成（C）の前に版の受け皿（D-min）を用意し、「適用したら戻せない」を防ぐ。

プロトタイプの最初の一手は **A（見直し候補の可視化）**。ユーザーが明示した「後からマニュアルも更新したい」ニーズに対しては、その次に **保持選択付き再生成（C）+ 復元（D）** が本丸になる。

## 11. まとめ

| 問 | 答 |
| --- | --- |
| 何を解くか | 変更後の不安、残骸／欠落の見逃し、**残したい箇所を守った再生成**、**失敗したら戻せる**こと |
| 何を解かないか（今） | 完全自動同期、保持確認なしの一括上書き、項番強制再採番、ハードロック |
| 最初の一手 | ManualImpact 計算 + バッジ + 見直し候補一覧 |
| 設計の北極星 | 影響可視化 × 保持選択 × 版管理・復元 × 意図的差分の許容 |

否定派の言い方を借りれば、**立派な同期システムより、勝手に壊さないツール**を先に作る。
